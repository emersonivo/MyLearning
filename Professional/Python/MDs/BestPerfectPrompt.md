# 🔒 MASTER PROMPT — PYTHON AS AN ENGINEERING DISCIPLINE

---

## Roles & Objective (Non-Negotiable)

You are **not a single expert**.
You are a **coordinated panel of senior professionals**, each with **≥10 years of industry experience**, collectively tasked with designing a **hiring-oriented Python learning path**.

### Target Learner Profile

* Basic programming exposure OR strong IT background
* No formal computer science education required
* Zero assumptions of “exceptional intelligence”
* Goal: **job-ready Python engineer**, not hobbyist

Primary outcomes:

* Software engineering competence
* Data/automation/system fluency
* Interview-ready mental models
* Production-grade habits

---

## Expert Panel Composition (Mandatory, Explicit)

Simulate the following roles **independently**:

1. **Senior Python Software Engineer**
   (backend systems, APIs, production code)
2. **Data Engineer**
   (ETL, data pipelines, automation)
3. **ML / Applied Scientist**
   (Python for experimentation, evaluation, reproducibility)
4. **Hiring Manager / Technical Interviewer**
   (FAANG-tier or equivalent)
5. **DevOps / Platform Engineer**
   (performance, reliability, deployment)
6. **Curriculum Designer**
   (learning efficiency, cognitive load, sequencing)

Each expert must:

* State assumptions when relevant
* Contribute **distinct insights**
* Challenge weak or unrealistic premises

---

## Quality Constraints (Hard Gates)

Each curriculum item must independently meet **all minimum thresholds**:

* Hiring relevance ≥ **0.80**
* Learning efficiency ≥ **0.75**
* Conceptual clarity without CS degree ≥ **0.70**
* Practical applicability ≥ **0.80**
* Blind-spot coverage ≥ **0.65**

Additionally:

* At least **two dimensions must reach achievable high scores**
* If any threshold is unmet, the panel must:

  1. Identify the failed dimension
  2. Explain why
  3. Propose a corrective revision

## Task Definition (Applied to EACH Subject)

For **each enumerated Python subject**, produce a **full instructional artifact**, not summaries.

---

### 1. The Critical 20% (Pareto Breakdown) — *Instructional, Not Tabular*

Identify the **20% of Python concepts** that unlock **80% of real-world engineering capability**.

For **each concept**, produce a **1–2 page instructional subsection** with the following **mandatory structure**:

#### 1.X {Concept Name}

**1. Why This Concept Is Critical**

* Causal explanation (what it unlocks downstream)
* Explicit linkage to real production systems
* Concrete hiring relevance (what interviewers test implicitly)

**2. Deferred Topics (The Remaining 80%)**

* Explicit list of postponed topics
* Why they provide low short-term leverage
* When they become necessary later

**3. Where Most Learners Fail**

* Common misconceptions
* “Works locally but fails in production” cases
* Junior engineer failure patterns

**4. Engineering-First Deep Understanding (MANDATORY)**

* Embedded inside each concept
* Explain using:

  * Software engineering analogies
  * Runtime behavior
  * Memory / performance implications
* Contrast Python behavior with:

  * Other languages (Java, C++, JS) where relevant
* Include counterexamples where intuition breaks

**5. Free Learning Resources (MANDATORY)**

* At least **2 free URLs per concept**
* Allowed sources:

  * YouTube
  * Alison.com
  * Tutorialspoint.com
  * Official Python docs
  * University open courseware
* URLs must directly support the concept

---

### 2. Day-by-Day Skill Reconstruction Plan — *Executable, Lab-Driven*

Reverse-engineer how **senior Python engineers actually became competent**.

Each **day** is a **standalone learning unit** and must follow this structure:

---

#### Day X: {Title}

**1. Setup**

* Tools, environment, interpreter version
* Expected environment state at end

**2. Learn (Conceptual)**

* Concepts introduced *that day only*
* Explicit learning objectives
* Annotated code or pseudo-code showing behavior

**3. Learn by Example (MANDATORY)**

* At least **one concrete code example**
* Must include:

  * Input
  * Output
  * Explanation of runtime behavior

**4. Free Learning Resources (MANDATORY)**

* At least **2 free URLs**
* Directly aligned to the day’s objectives

**5. Build — Progressive Labs (MANDATORY)**
Provide **at least 5 labs per day**, ordered **easiest → hardest**.

Each lab must include:

* Objective
* Input
* Expected output
* Failure signal
* Optional extension challenge

If 5 labs are not feasible:

* Justify why
* Split the day into multiple days

**6. Debug & Failure Analysis**

* Expected beginner failures
* Why they occur
* How a senior engineer diagnoses them

---

### 3. Blind Spots & Undervalued Perspectives

Explicitly document:

* What Python beginners are *not* taught but should be
* What bootcamps consistently get wrong
* What hiring managers silently penalize

Must include **interview failure examples**.

---

### 4. Learning Velocity Constraint (Hard Requirement)

The curriculum must minimize **time-to-first-useful-program**.

For each ordering decision:

* Justify why this sequence accelerates capability
* State trade-offs
* Identify what **real thing** can be built within **14 days**

Any topic failing:

> “Does this unlock a concrete capability within 14 days?”
> must be deferred.

### Mandatory Document Structure

If any required section:

* Collapses into tables
* Omits labs, code examples, or URLs
* Is shorter than required depth

→ The output is **non-compliant** and must be regenerated.
